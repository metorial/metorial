# Slates Specification for Kubernetes

## Overview

Kubernetes is an open-source container orchestration platform that automates the deployment, scaling, and management of containerized applications. The Kubernetes API lets you query and manipulate the state of objects in Kubernetes. All operations and communications between components, and external user commands are REST API calls that the API Server handles.

## Authentication

Kubernetes supports multiple authentication methods, and clusters can enable several simultaneously. All Kubernetes clusters have two categories of users: service accounts managed by Kubernetes, and normal users.

### X.509 Client Certificates

Any user that presents a valid certificate signed by the cluster's Certificate Authority is considered authenticated. This requires:

- The cluster's CA certificate (typically found in the kubeconfig file)
- A client certificate signed by the cluster CA
- The corresponding client private key

The username is extracted from the certificate's Common Name (CN) field, and groups from the Organization (O) field.

### Bearer Token Authentication

Another way to authenticate API requests is through using a bearer header containing a valid Service Account JWT token. Tokens are passed via the `Authorization: Bearer <token>` HTTP header. Service account tokens are automatically created and mounted into pods. Tokens can also be created manually via the TokenRequest API.

### OpenID Connect (OIDC)

OpenID Connect (OIDC) is an extension of the OAuth 2.0 authentication standard. The OIDC authentication layer can validate users if they present an unexpired and valid JWT token without invoking a third-party service such as GitHub or other auth providers. The API server must be configured with the OIDC provider's issuer URL, client ID, and optionally claim mappings for username and groups.

### Webhook Token Authentication

Webhook Token Authentication allows you to configure authorization with third-party generated tokens. In this authentication mode, API requests contain a token generated by external services. Since the token does not come from Kubernetes and Kubernetes cannot validate them with OIDC mechanisms, the cluster must call a remote service to check their trustworthiness. Therefore, the API server will validate each token received by calling a third-party service validation endpoint.

### Authenticating Proxy

The Kubernetes API server can identify users based on request value headers. These values are set by an authenticating proxy. Authentication information, such as usernames and namespaces, can be pulled from these headers in order to determine whether a request is valid.

### Kubeconfig File

In practice, most clients authenticate using a **kubeconfig** file (typically at `~/.kube/config`) which bundles the API server URL, credentials (certificates, tokens, or OIDC configuration), and cluster context. This is the standard way to provide connection details to API clients and is what tools like `kubectl` use.

## Features

### Workload Management

Workloads are objects you use to manage and run your containers on the cluster. This includes creating, updating, scaling, and deleting:

- **Pods** — the smallest deployable unit containing one or more containers
- **Deployments** — for managing stateless applications with rolling updates and rollback
- **StatefulSets** — for stateful applications requiring stable identifiers and persistent storage
- **DaemonSets** — for running a copy of a pod on every (or selected) node
- **Jobs and CronJobs** — for run-to-completion and scheduled batch workloads
- **ReplicaSets** — for maintaining a stable set of pod replicas

### Service Discovery and Load Balancing

Discovery & LB resources are objects you use to "stitch" your workloads together into an externally accessible, load-balanced Service. This includes:

- **Services** — provide a stable IP and DNS name for a set of pods
- **Ingress** — HTTP(S) routing rules to expose services externally
- **Endpoints / EndpointSlices** — track the network addresses of pods backing a service

### Configuration and Storage

Config & Storage resources are objects you use to inject initialization data into your applications, and to persist data that is external to your container. This includes:

- **ConfigMaps** — key-value configuration data for pods
- **Secrets** — sensitive data such as passwords, tokens, or certificates
- **PersistentVolumes and PersistentVolumeClaims** — provisioning and requesting persistent storage
- **StorageClasses** — defining classes of storage for dynamic provisioning

### Namespace and Access Control

Cluster resources objects define how the cluster itself is configured; these are typically used only by cluster operators. This includes:

- **Namespaces** — logical partitioning of cluster resources for multi-tenancy
- **RBAC resources** (Roles, ClusterRoles, RoleBindings, ClusterRoleBindings) — defining who can do what on which resources
- **ServiceAccounts** — identities for processes running in pods
- **NetworkPolicies** — firewall rules controlling pod-to-pod traffic

### Cluster Administration

Manage the cluster infrastructure itself:

- **Nodes** — view and manage worker nodes
- **ResourceQuotas and LimitRanges** — control resource consumption per namespace
- **PriorityClasses** — define scheduling priority for pods
- **Leases** — coordination and leader election primitives

### Autoscaling

Scale workloads dynamically based on metrics:

- **HorizontalPodAutoscaler** — adjusts the number of pod replicas based on CPU, memory, or custom metrics
- **VerticalPodAutoscaler** (if installed) — adjusts resource requests/limits for containers

### Custom Resource Definitions (CRDs)

The Kubernetes API is also extensible. You can add your own APIs and register them with Kubernetes. CRDs allow users to define and manage custom resource types, enabling operators and controllers to extend Kubernetes functionality for domain-specific use cases.

## Events

Kubernetes provides a built-in **Watch** mechanism on its API for real-time change streaming.

### Watch API

Kubernetes supports efficient change notifications on resources via watches: in the Kubernetes API, watch is a verb that is used to track changes to an object in Kubernetes as a stream. The server will return all changes (creates, deletes, and updates) that occur after the supplied resourceVersion. This allows a client to fetch the current state and then watch for changes without missing any updates.

Watches can be opened on any resource type (Pods, Deployments, Services, ConfigMaps, Secrets, custom resources, etc.) by adding a `?watch=true` query parameter to a list endpoint. The response is a long-lived HTTP stream delivering events of type `ADDED`, `MODIFIED`, or `DELETED` as they occur.

- **Parameters**: `resourceVersion` (to resume from a specific point), `labelSelector` and `fieldSelector` (to filter which resources to watch), `timeoutSeconds` (to control stream duration).
- **Scope**: Watches can be scoped to a single namespace or cluster-wide.
- The Kubernetes API server generally only retains a limited history of resourceVersions. If a client's watch connection is down for too long, it might fall behind the available resourceVersion window and require a full resync.

### Admission Webhooks

Admission webhooks are HTTP callbacks that receive admission requests and do something with them. You can define two types of admission webhooks, validating admission webhook and mutating admission webhook. Mutating admission webhooks are invoked first, and can modify objects sent to the API server to enforce custom defaults. After all object modifications are complete, and after the incoming object is validated by the API server, validating admission webhooks are invoked and can reject requests to enforce custom policies.

- **MutatingWebhookConfiguration** — intercepts and optionally modifies resources before they are persisted
- **ValidatingWebhookConfiguration** — intercepts and validates resources, rejecting those that violate custom policies
- **Parameters**: Rules specify which resources, API groups, API versions, and operations (CREATE, UPDATE, DELETE, CONNECT) trigger the webhook. A `namespaceSelector` and `objectSelector` can further filter scope.
- Admission webhooks require a running HTTPS server endpoint accessible from the API server.
