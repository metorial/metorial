# Slates Specification for PagerDuty

## Overview

PagerDuty is an incident management platform that helps operations teams detect, triage, and resolve infrastructure and application incidents. The PagerDuty REST API provides 200+ endpoints for users to programmatically access objects and workflows in the PagerDuty platform. It offers both a REST API for resource management and an Events API for ingesting machine-generated events from monitoring tools.

## Authentication

PagerDuty supports three authentication methods:

### 1. REST API Keys (Token-based)

There are two types of REST API keys:

- **General Access API Keys (Account-level):** Can be generated by Admins and Account Owners. API key has full admin permissions. Optionally check Read-only API Key if you'd like the key to only make GET calls.
- **Personal API Keys (User-level):** If your account has Advanced Permissions, users can create personal REST API keys. Requests made using personal REST API keys are restricted to the user's permissions. Any client request for an operation that the user is not permitted to perform will result in a 403 Forbidden response.

Both types of REST API keys are 20-character strings. They are passed in the `Authorization` header as: `Authorization: Token token=YOUR_API_KEY`.

### 2. OAuth 2.0 (Scoped OAuth)

PagerDuty supports OAuth 2.0 with two variants:

- **Scoped OAuth (recommended):** Allows granular read or write access to PagerDuty resources like incidents, services, users, with other benefits. Only account admins are able to create applications using the Scoped OAuth 2.0 features.
- **Classic User OAuth:** Existing OAuth client that allows apps to act on behalf of users, with read or write to all PagerDuty resources.

**OAuth Endpoints:**

- Authorization: `https://app.pagerduty.com/oauth/authorize`
- Token: `https://app.pagerduty.com/oauth/token`

**Scoped OAuth also supports Client Credentials flow:**

- Tokens will be requested from `https://identity.pagerduty.com/oauth/token` using the credentials you received when the app was created.
- The other required data are your region (US or EU) and subdomain (youraccount.pagerduty.com).

**Scopes** follow the pattern `{resource}.{access}` (e.g., `incidents.read`, `services.write`). Each object in the PagerDuty REST API will have at least one scope, read, and many objects will also have write. You'll be able to tune your apps to have only the access they are required to have. A region/subdomain prefix is required for scoped tokens, e.g., `as_account-us.companysubdomain`.

The bearer token will only be valid for 24 hours (user OAuth) or Scoped OAuth tokens will have a 30-day expiration. Tokens are passed as: `Authorization: Bearer YOUR_ACCESS_TOKEN`.

### 3. Events API Integration Keys

The Events API is available to all customers, and any user with a Manager role or above can generate an integration key. API keys for the Events API are 32-character strings. They are associated with service-level integrations. These are used exclusively for sending events (alerts and changes) to PagerDuty, not for REST API calls.

**Note:** PagerDuty has separate API base URLs for US and EU regions. The default is `https://api.pagerduty.com` (US) with EU accounts using `https://api.eu.pagerduty.com`.

## Features

### Incident Management

Create, view, update, acknowledge, and resolve incidents. Incidents can be triggered, acknowledged, or resolved, and are assigned to users based on escalation policies. Incidents can be triggered, acknowledged, or resolved, and are assigned to a User based on the Service's Escalation Policy. Supports merging incidents, adding notes/annotations, snoozing, reassignment, managing responders, posting status updates, and setting priority.

### Event Ingestion (Events API)

The Events API v2 is a highly reliable, highly available asynchronous API that ingests machine events from monitoring tools and other systems like code repositories, observability platforms, automated workflow tools, and configuration management systems. Supports two event types: **Alert events** (trigger/acknowledge/resolve incidents) and **Change events** (provide deployment/change context to responders). Events use a routing key tied to a service integration.

### Service Management

Services represent applications, components, or infrastructure that teams manage. You can create, update, delete, and configure services including their alert creation settings, auto-resolve timeouts, acknowledgement timeouts, urgency rules, and support hours. Services contain integrations that define how events flow in.

### Escalation Policies

Define multi-level escalation rules that determine who gets notified when incidents occur and how they escalate if unacknowledged. Each level can target users or schedules. There is a limit of 1 team per Escalation Policy.

### On-Call Schedules

A Schedule determines the time periods that Users are On-Call. Only On-Call Users are eligible to receive Notifications from Incidents. Supports creating rotation layers, overrides, and querying who is currently on-call.

### User Management

Create, update, and delete user accounts. Manage user contact methods (email, phone, SMS, push), notification rules (how and when users are contacted), and user roles/permissions.

### Teams

A Team is a collection of Users and Escalation Policies that represent a group of people within an organization. Teams can be used throughout the API and PagerDuty applications to filter information to only what is relevant for one or more teams.

### Event Orchestration

Configure event processing rules that determine how incoming events are routed, transformed, and handled. Supports conditions, suppression, severity mapping, and automated actions.

### Business Services

Model business-level services that depend on technical services. Used for impact analysis and stakeholder communication during incidents.

### Analytics & Reporting

Access incident analytics data including metrics aggregated by services, teams, and time periods. Retrieve raw incident data for custom reporting.

### Tags

A Tag can be assigned to Escalation Policy, Team or User, and searches for those objects can be filtered to retrieve those with specific tags.

### Maintenance Windows

Maintenance windows are specified to start at a certain time and end after they have begun. Once started, a Maintenance Window cannot be deleted; it can only be ended immediately to re-enable the Service. Used to temporarily disable incident creation on services during planned maintenance.

### Notification Management

A Notification is created when an Incident is triggered or escalated and an Alert is sent to a User. Notifications are messages containing the details of the Incident, and can be through SMS, email, phone calls, and push notifications. Notifications cannot be created directly through the API; they are a result of other actions. The API provides read-only access to the Notifications generated by PagerDuty.

### Incident Workflows

Automate sequences of actions that can be triggered when incidents are created or meet certain conditions. Replaces the deprecated Response Plays feature.

### Custom Fields

Define custom fields on incidents and services to capture organization-specific metadata.

### Audit Trail

Access audit records that track changes made to resources across the PagerDuty account.

### Extensions & Integrations

Manage integrations on services and extensions (third-party tool connections like Slack, Jira, etc.).

## Events

PagerDuty supports webhooks (V3 Webhook Subscriptions) that send HTTP POST callbacks when events occur in the account.

Webhooks allow you to receive HTTP callbacks when significant events happen in your PagerDuty account, for example, when an incident triggers, escalates or resolves.

Webhook subscriptions can be scoped using filters: There are currently three types of filters that can be applied to webhook subscriptions: service_reference, team_reference and account_reference.

Payloads are signed with HMAC for verification, and custom headers can be added to webhook deliveries.

### Incident Events

Events related to the lifecycle and updates of incidents:

- `incident.triggered` — A new incident is triggered
- `incident.acknowledged` — An incident is acknowledged
- `incident.unacknowledged` — An incident acknowledgement expires/is removed
- `incident.resolved` — An incident is resolved
- `incident.escalated` — An incident is escalated to a new level
- `incident.delegated` — An incident is reassigned via delegation
- `incident.reassigned` — An incident is reassigned to a different user
- `incident.reopened` — A resolved incident is reopened
- `incident.annotated` — A note is added to an incident
- `incident.priority_updated` — An incident's priority is changed
- `incident.responder.added` — A responder is added to an incident
- `incident.responder.replied` — A responder replies to a request
- `incident.status_update_published` — A status update is published on an incident

### Service Events

Events related to changes in service configuration:

- `service.created` — A new service is created
- `service.updated` — A service is updated
- `service.deleted` — A service is deleted
