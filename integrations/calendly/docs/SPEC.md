# Slates Specification for Calendly

## Overview

Calendly is a scheduling platform that allows users to create event types, share booking links, and let invitees self-schedule meetings. It handles calendar sync, availability management, reminders, and routing forms for lead qualification. The API (v2) provides access to scheduling data, availability, event management, organization administration, and programmatic booking via the Scheduling API.

## Authentication

Calendly supports two authentication methods:

### 1. Personal Access Tokens

For internal applications that only your team will use, authenticate with personal access tokens. Tokens are generated from the Calendly account's Integrations page under "API & Webhooks." Calendly does not display or store personal access tokens in your account. After generation, they're unretrievable.

Tokens are passed as a Bearer token in the `Authorization` header:

```
Authorization: Bearer <personal_access_token>
```

The scope of available data depends on whether the token is generated by a user or an admin/owner. A user role token only allows access to data linked to that specific user, including webhooks for their own events. An admin/owner token provides broader access, allowing API calls to manage data across the entire organization.

### 2. OAuth 2.1

For public applications that many Calendly users will use, authenticate with OAuth. Calendly uses OAuth 2.1 with the authorization code grant type.

**Endpoints:**

- Authorization URL: `https://auth.calendly.com/oauth/authorize`
- Token URL: `https://auth.calendly.com/oauth/token`

**Flow:**

1. Initiate the OAuth authorization flow using your application's `client_id` and redirect your user to the authorization endpoint. Once the Calendly user grants access, their browser is redirected to your application's specified `redirect_uri` and the authorization code is passed inside the `code` query parameter.
2. Exchange the authorization code for an access token by POSTing to the token endpoint with `grant_type=authorization_code`, `client_id`, `client_secret`, `code`, and `redirect_uri`.
3. Access tokens expire after 2 hours. Use a refresh token to get a new access token.

**Setup:** Sign up for a Calendly developer account (separate from your Calendly user account) using your GitHub or Google account. Create an OAuth application and choose between Sandbox and Production environments. You will receive a `client_id` and `client_secret`.

Calendly uses granular scopes for newly created OAuth apps and personal access tokens. This integration requests the read/write scopes needed for users, organizations, event types, availability, scheduled events, scheduling links, routing forms, and webhook registration. Access is still constrained by the authenticated user's Calendly role and subscription plan.

## Features

### Scheduled Events Management

Retrieve and list scheduled events for a user or across an organization. Cancel existing events. View event details including invitee information, questions and answers, and tracking/UTM data. The API does not support scheduling events directly through the standard events endpoints — for that, use the Scheduling API (see below).

### Scheduling API (Programmatic Booking)

The Scheduling API makes it possible to create new Calendly events programmatically. Using the Create Event Invitee endpoint, users can book meetings on behalf of invitees directly from apps and workflows — no redirects, iframes, or Calendly UI required. Requires retrieving available times first, then posting invitee details with a chosen time slot. Calendly customers are required to be on a paid plan to access or use applications calling the Scheduling API.

### Event Types

Retrieve and list event types for a user or organization. Event types represent the different meeting templates (e.g., "30-min Demo", "Onboarding Call") with their configured durations, locations, and custom questions. Calendly's API also exposes create/update event type endpoints, but this integration currently focuses on reading event types and booking against existing event types.

### Availability

Users can retrieve information about the availability of users in their organization, including programmatically calculating total hours available for a given user for a given period of time. Query available time slots for a specific event type within a date range, retrieve user availability schedules and date overrides, and check user busy times. The API does not support setting or managing availability.

### Scheduling Links

Create single-use scheduling links for event types. Calendly also exposes a separate share endpoint for customized single-use links; this integration currently supports the standard scheduling link endpoint.

### Invitee Management

List and retrieve invitees for scheduled events, including their contact information, responses to custom questions, and UTM tracking data. Mark invitees as no-shows.

### Organization Management

Retrieve organization details, list organization memberships, invite users to an organization, and remove members. Access is restricted based on user role (admin/owner required for most organization-level operations).

### Routing Forms

Retrieve routing forms and their submissions. Routing forms direct visitors to specific people or destinations based on factors like industry, company size, and interests. The API provides read access to form configurations and submission data.

### User Management

Retrieve user profiles and information, including scheduling URLs and organization membership details.

## Events

Calendly supports webhooks for real-time notifications. Access to webhooks is reserved for members on paid premium subscriptions and above. Webhook subscriptions can be scoped to a single user or an entire organization.

### Invitee Created (`invitee.created`)

Fires when a new event is scheduled by an invitee. The payload includes event details, invitee information, and tracking data. Can be scoped to user or organization level.

### Invitee Canceled (`invitee.canceled`)

Fires when an event is canceled by an invitee. Rescheduled events trigger both a cancellation of the original and creation of the new event. Can be scoped to user or organization level.

### Routing Form Submission (`routing_form_submission.created`)

Fires anytime someone submits a routing form, whether they book or not. The only allowed subscription scope for routing form submissions is organization.

**Configuration options for all webhook types:**

- `url`: The endpoint URL to receive webhook payloads.
- `scope`: Either `user` (events for a specific user) or `organization` (events across the entire organization). Routing form submissions only support organization scope.
- `organization`: The organization URI (required for both scopes).
- `user`: The user URI (required for user scope).
- Webhook signing keys are provided when creating an OAuth application, enabling payload signature verification.
